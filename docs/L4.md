# L4 SPEC: Maintainer’s Cabinet

## Scope

This is the **L4 version** of the MaaS project.

The L3 version proved a working GitHub-native agent team. The L4 version should feel like a **serious pre-production maintainer operations system**:

* Real GitHub output on real repos.
* Manager-led dynamic task planning.
* Persistent project memory.
* Automated eval pipeline.
* Useful observability with trace trees, cost, latency, and agent/tool breakdowns.
* Guarded autonomy for low-risk maintainer work.
* Clean management UI that a non-engineer can operate with a walkthrough.

The goal is **not yet full L5 emergent organization**. Do not build arbitrary sub-agent spawning, full self-adjusting teams, or production-grade autonomous release/merge behavior yet.

The project should directly address the maintainer toil described in the uploaded maintainer-day analysis: issue triage, duplicate/support overload, PR review burden, docs drift, release engineering, community friction, and security-sensitive interruptions. 

---

# 1. L4 product definition

## Name

**Maintainer’s Cabinet**

## One-liner

A GitHub-native multi-agent maintainer operations system that triages issues, reviews PRs, detects docs and release impact, remembers project decisions, and explains every action through an auditable trace.

## L4 positioning

At L4, this should be more than a demo bot.

It should be:

> A guarded-autonomy GitHub App where a manager agent dynamically coordinates specialist agents, takes safe actions on real GitHub surfaces, persists project context across tasks, and blocks regressions through automated evals.

# 2. L4 rubric target

## Working product: L4

Target behavior:

* Runs on a real GitHub repo.
* Produces real GitHub output.
* Handles real issues and PRs, not only canned examples.
* Safe actions happen automatically.
* Higher-risk actions are proposed for maintainer approval.
* Maintainer can use it in daily workflow.

L4 acceptable outputs:

* Labels added.
* Comments posted.
* PR check runs created.
* PR reviews posted.
* Draft release checklist created.
* Draft release notes created.
* Docs-impact comments posted.
* Stale `needs-info` issues proposed for closure.
* Daily maintainer briefing issue updated.
* Persistent project memory updated after approval.

Not required until L5:

* Fully autonomous PR merging.
* Fully autonomous release publishing.
* Emergent agent creation.
* Production-grade reliability under heavy repo traffic.
* Self-adjusting agent org.

---

## Agent org structure: L4

Target behavior:

* Manager agent plans subtasks dynamically.
* Fixed specialist agents exist.
* Manager chooses which agents to call based on event type, repo policy, risk, and prior results.
* Manager reviews outputs before GitHub actions execute.
* Manager resolves simple conflicts.

L4 agent set:

* Cabinet Manager
* Triage Agent
* PR Review Agent
* Docs Agent
* Release Agent
* Community Agent
* Memory Agent
* Eval Agent

L4 does **not** require arbitrary new agent roles created on the fly. That is L5.

---

## Observability: L4

Target behavior:

* Trace tree across agents.
* Shows who called whom.
* Shows each tool call.
* Shows token and cost per step.
* Shows latency per step.
* Filter runs by repo, agent, task, status, cost, and GitHub target.
* Shows final GitHub mutations.

L4 does **not** require production-grade alerting, side-by-side replay diffing, or senior-engineer-trusted prod debugging at L5 level.

---

## Evaluation and iteration: L4

Target behavior:

* Automated eval pipeline exists.
* Runs on prompt or policy changes.
* Runs in CI.
* Fails if quality drops.
* Eval results are visible in dashboard and GitHub check.

This should be one of the clearest L4 differentiators.

---

## Agent memory: L4

Target behavior:

* Persistent project memory across tasks.
* Cabinet remembers prior maintainer decisions.
* Cabinet remembers repo policy.
* Cabinet remembers duplicate clusters.
* Cabinet remembers release conventions.
* Cabinet remembers docs structure.
* Cabinet remembers recurring support topics.

L4 memory does not need full hierarchical working/episodic/semantic L5 architecture, but it must be real and inspectable.

---

## Cost and latency: L4

Target behavior:

* Typical task completes in **1–5 minutes** or under **$0.10–$0.50**.
* Cost and latency are visible.
* There are soft budgets.
* Manager can skip optional agents if budget is at risk.

Hard production optimization is L5.

---

## Management UI: L4

Target behavior:

* Clean web UI.
* Guided onboarding.
* Non-engineer can operate with one walkthrough.
* Maintainer can approve/reject proposed actions.
* Maintainer can edit policy.
* Maintainer can inspect memory.
* Maintainer can view eval results.

No-code custom agent creation is L5.

# 5. Core product behavior

A repo maintainer installs the GitHub App.

The app reads `.github/cabinet.yml`, indexes the repo, creates labels, and starts processing GitHub events.

For every incoming issue or PR, the Cabinet Manager:

1. Builds a work packet.
2. Retrieves relevant project memory.
3. Plans which agents to invoke.
4. Calls the selected agents.
5. Detects conflicts.
6. Chooses safe GitHub actions.
7. Sends risky actions to approval queue.
8. Executes allowed actions.
9. Records a trace.
10. Updates memory when appropriate.
11. Updates the daily briefing issue.

# 11. Agents

## 11.1 Cabinet Manager

The Cabinet Manager is the L4 orchestrator.

### Responsibilities

* Classify incoming event.
* Load repo config.
* Retrieve relevant memory.
* Decide which agents to call.
* Create an execution plan.
* Pass structured work packets between agents.
* Review agent outputs.
* Resolve conflicts.
* Decide safe actions.
* Send risky actions to approval queue.
* Execute approved actions through GitHub tools.
* Record trace.
* Update memory candidates.

### L4 dynamic planning

The Manager should not use only static routing.

Example:

```json
{
  "run_type": "pull_request_review",
  "target": "PR #42",
  "risk": "medium",
  "selected_agents": [
    "pr_review",
    "docs",
    "release",
    "community"
  ],
  "skipped_agents": [
    {
      "agent": "triage",
      "reason": "Not relevant to pull_request event"
    }
  ],
  "plan": [
    "Summarize changed files",
    "Run PR Review Agent",
    "Run Docs Agent if public behavior changed",
    "Run Release Agent if user-facing behavior changed",
    "Run Community Agent on final public comment",
    "Create check run",
    "Post PR comment",
    "Queue request-changes recommendation for approval if blocking findings exist"
  ]
}
```

### Manager output

```ts
type ManagerDecision = {
  run_type:
    | "issue_triage"
    | "issue_update"
    | "pr_review"
    | "ci_failure"
    | "release_plan"
    | "daily_briefing"
    | "stale_scan";

  selected_agents: string[];

  actions_to_execute: ActionProposal[];

  actions_requiring_approval: ActionProposal[];

  rejected_actions: {
    action: ActionProposal;
    reason: string;
  }[];

  conflicts: {
    agents: string[];
    issue: string;
    resolution: string;
  }[];

  memory_candidates: MemoryCandidate[];

  public_summary: string;
};
```

### L4 capabilities

* Detect missing bug fields.
* Find likely duplicates.

* Recognize docs issues.
* Recognize security-sensitive issues.

* Queue duplicate close for approval.
* Mark issue ready for maintainer.

* Directly close issues.
* Publicly confirm vulnerabilities.
* Accuse users.
* Decide roadmap.

# 12. Workflows

## 12.1 New issue triage

### Trigger

```text
issues.opened
/cabinet triage
```

### Flow

1. Ingest webhook.
2. Verify signature.
3. Store event.
4. Manager creates run.
5. Load `.github/cabinet.yml`.
6. Retrieve similar issues.
7. Retrieve relevant memory.
8. Manager selects Triage + Community.
9. Triage classifies issue.
10. Community rewrites public comment.
11. Manager decides actions.
12. Execute allowed actions:

    * Add labels.
    * Post comment.
13. Queue risky actions:

    * Close duplicate.
    * Route security issue.
14. Store trace.
15. Update daily briefing.
16. Create memory candidate if useful.

### Example action

```markdown
Thanks for reporting this. I need a few details before this is actionable:

- Package version
- Environment
- Minimal reproduction
- Expected behavior
- Actual behavior

This may be related to #42, but the details above are needed to confirm.

_Cabinet run: `run_01HX9` · Agents: Triage, Community · Trace: dashboard_
```

---

## 12.2 Needs-info follow-up

### Trigger

```text
issue_comment.created
```

### Flow

1. Detect whether issue has `cabinet:needs-info`.
2. Extract new details from comment.
3. Triage checks whether required fields are now present.
4. If complete:

   * Remove `cabinet:needs-info`.
   * Add `cabinet:ready-for-maintainer`.
   * Post concise summary.
5. Store trace.
6. Update memory if a pattern emerges.

### Example output

```markdown
Thanks. This issue now has the required reproduction details.

Summary for maintainers:
- Version: 2.4.1
- Environment: macOS 14, Node 20
- Reproduction: `cabinet-demo parse --strict input.json`
- Actual behavior: parser exits with code 1
- Expected behavior: parser should accept escaped spaces

I marked this as ready for maintainer review.

_Cabinet run: `run_01HXA`_
```

---

## 12.3 Duplicate candidate workflow

### Trigger

```text
issues.opened
/cabinet triage
```

### Flow

1. Triage finds similar issue above threshold.
2. Manager checks policy.
3. If confidence < exact threshold:

   * Add `cabinet:possible-duplicate`.
   * Comment with possible duplicate.
4. If confidence >= exact threshold:

   * Queue duplicate close for maintainer approval.
   * Do not close automatically by default.

### Example approval queue item

```text
Proposed action:
Close issue #81 as duplicate of #42.

Confidence: 0.93
Evidence:
- Same stack trace.
- Same macOS version.
- Same failing command.
- Issue #42 is still open and active.

Requires approval because close_issues=approval_required.
```

---

## 12.4 PR review workflow

### Trigger

```text
pull_request.opened
pull_request.synchronize
/cabinet review
```

### Flow

1. Fetch PR metadata.
2. Fetch changed files and patch.
3. Fetch CI state.
4. Retrieve review memory.
5. Manager selects PR Review Agent.
6. If public behavior changed, Manager selects Docs Agent.
7. If user-facing change, Manager selects Release Agent.
8. Community Agent reviews final public comment.
9. Manager creates or updates check run.
10. Manager posts PR comment or PR review.
11. Manager queues request-changes if needed.
12. Store trace.

### Example check run

```markdown
## Cabinet Review

Risk: medium

Findings:
- Parser behavior changed without regression tests.
- Docs may need updates for CLI behavior.
- Release note likely needed.

Suggested next steps:
- Add parser regression tests.
- Update `docs/cli.md`.
- Add release note under Changed.

Queued for maintainer:
- Request changes recommendation.

Trace: `run_01HXB`
```

## 12.5 Docs impact workflow

### Trigger

```text
pull_request.opened
pull_request.synchronize
/cabinet docs-impact
```

### Flow

1. Docs Agent checks changed files.
2. Finds public API, CLI, config, or behavior changes.
3. Maps to docs files.
4. Adds `cabinet:docs-needed`.
5. Comments affected docs.
6. Drafts patch.
7. Queues docs patch PR for approval.

### Example comment

```markdown
This PR appears to change user-visible CLI behavior.

Docs likely affected:
- `docs/cli.md`: document `--strict-mode`
- `README.md`: update quickstart example if strict mode becomes recommended

I drafted a docs patch for maintainer approval.

_Cabinet run: `run_01HXC`_
```

## 12.6 Release planning workflow

### Trigger

```text
/cabinet release-plan
scheduled weekly release scan
milestone due soon
```

### Flow

1. Fetch last release or tag.
2. Fetch merged PRs since last release.
3. Fetch open release-blocker issues.
4. Fetch CI status on default branch.
5. Release Agent groups changes.
6. Docs Agent checks migration/docs blockers.
7. Manager creates release checklist issue.
8. Draft release notes are included.
9. Draft release creation is queued for approval.

### Example issue

```markdown
# Cabinet Release Checklist

Recommended version impact: minor

## Blockers

- [ ] PR #42 changes CLI behavior but docs are not updated.
- [ ] Issue #58 is still labeled release-blocker.

## Draft release notes

### Added
- Add strict mode for config validation.

### Fixed
- Fix parser crash on escaped spaces.
```

# 20. Conflict handling

L4 should explicitly show conflicts.

## Conflict types

### Triage vs Community

```text
Mark likely duplicate and close.
```

```text
Avoid blunt closing; ask for clarification.
```

```text
Add possible-duplicate label, comment gently, queue close for approval.
```

---

### PR Review vs Release

Release wants:

```text
This PR is needed for next release.
```

PR Review says:

```text
Missing tests; risky path.
```

Manager resolution:

```text
Mark as release blocker. Do not recommend merge.
```

---

### Docs vs PR Review

PR Review says:

```text
Low code risk.
```

Docs says:

```text
User-facing behavior changed.
```

Manager resolution:

```text
Add docs-needed and include in check run.
```

---

### Security vs Community

Community says:

```text
Ask for details.
```

Triage says:

```text
Potential vulnerability.
```

Manager resolution:

```text
Post neutral public comment, flag for maintainer, avoid exploit details.
```

# 28. What to cut if time is tight

Cut these first:

* Opening docs PRs.
* Creating draft releases.
* Discussions integration.
* Workflow log deep analysis.
* Multi-repo dashboard.
* Complex stale automation.
* Memory expiration.
* Edit-and-approve action UI.
* Fine-grained contributor permissions.

Do **not** cut:

* Dynamic Manager.
* Persistent memory.
* Approval queue.
* Trace tree.
* Cost/latency display.
* Eval CI gate.
* PR check run.
* Daily briefing issue.
* Real GitHub writes.

Those are the L4 scoring anchors.

# 29. Definition of done for L4

L4 is complete when the system can do this end-to-end:

* Install as GitHub App.
* Read `.github/cabinet.yml`.
* Triage real issues.
* Use persistent memory in a later run.
* Add labels and comments automatically.
* Queue close/request-changes/docs/release actions for approval.
* Review PRs through GitHub Check Runs.
* Detect docs impact.
* Detect release-note need.
* Create release checklist issue.
* Update daily briefing issue.
* Show agent trace tree with cost, tokens, latency, tools, and actions.
* Run automated evals in CI.
* Fail a prompt-change PR when evals regress.

That is the L4 version: strong, real, and defensible, without overreaching into L5 emergent autonomy.
