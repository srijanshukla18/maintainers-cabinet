# SPEC: The Open Source Maintainer’s Cabinet

## One-line pitch

**A GitHub-native multi-agent maintainer team that triages issues, reviews PRs, drafts releases, updates docs, manages community tone, remembers project context, and shows every decision in a debuggable trace.**

This should not look like “ChatGPT for GitHub.” It should look like a **maintainer operations layer** living inside GitHub.

The real problem is not “write code.” It is queue pressure: issue triage, AI-slop PRs, support disguised as bugs, documentation debt, release work, toxic interactions, and context loss. Your attached maintainer-day analysis makes that clear.

# 1. Winning strategy for the MaaS rubric

## Build for L5 in every category

Your project should visibly satisfy:

### Working product shipping real output

The system must autonomously complete real GitHub tasks:

* Label issues.
* Detect duplicates.
* Ask for missing reproduction details.
* Close or recommend closing low-quality issues.
* Review PRs with inline findings.
* Create a docs-fix PR.
* Draft release notes.
* Create a release readiness checklist.
* Post a daily maintainer briefing issue.
* Update persistent project memory after maintainer approval.

Do not only show a dashboard. Judges need to see outputs land on GitHub.

### Agent org structure

Use **five public cabinet agents plus one manager layer**.

The public story remains “five agents.” The architecture has a manager because the rubric rewards dynamic delegation.

Public cabinet:

* Triage Agent
* PR Review Agent
* Release Agent
* Community Agent
* Docs Agent

Manager layer:

* Cabinet Secretary

The Secretary receives GitHub events, decides which agents to invoke, resolves conflicts, tracks cost, records memory, and spawns temporary specialists when needed.

Temporary specialists:

* Security Scout
* Reproduction Builder
* Test Analyst
* Dependency Analyst
* Changelog Clerk
* Governance Clerk
* Tone Editor
* Docs Patch Writer

### Observability

Build your own run trace.

Every agent run gets:

* Run ID.
* GitHub event source.
* Agent tree.
* Tool calls.
* Memory reads.
* Proposed actions.
* Conflicts.
* Final GitHub mutations.
* Token count.
* Cost.
* Latency.
* Diff between intended and actual outputs.

A mentor should be able to debug a bad decision in under two minutes.

### Evaluation and iteration

Ship a built-in eval suite.

Use real saved GitHub cases:

* 20 historical issues.
* 10 duplicate issue pairs.
* 10 PRs.
* 5 docs mismatch cases.
* 5 release scenarios.
* 5 community tone cases.
* 5 adversarial AI-slop cases.

Every prompt or policy change triggers evals through GitHub Actions.

### Persistent memory

The Cabinet must remember:

* Project policies.
* Maintainer preferences.
* Label taxonomy.
* Prior duplicate decisions.
* Release cadence.
* Supported versions.
* Contributor trust history.
* Tone rules.
* Docs structure.
* “Never do this again” rules.

Memory must be visible and editable.

### Cost and latency

Show cost in the trace.

Targets:

* Issue triage: under 20 seconds.
* Duplicate detection: under 10 seconds after indexing.
* Small PR review: under 60 seconds.
* Release draft: under 60 seconds.
* Docs impact check: under 45 seconds.
* Average simple task: under $0.10.

### Management UI

A non-engineer must be able to:

* Install the GitHub App.
* Pick a repo.
* Choose autonomy level.
* Add repo policy.
* Add or edit an agent role.
* Run the agent on a past issue.
* Approve going live.

# 3. Core concept

Maintainer’s Cabinet installs as a GitHub App.

It listens to repo events, builds persistent project context, and runs a cabinet of agents that behave like a small maintainer team.

The agents do not merely “answer.” They take constrained, auditable GitHub actions.

The app should use GitHub-native surfaces first:

* Issues
* PR comments
* PR reviews
* Labels
* Milestones
* Check runs
* Draft releases
* Release notes
* GitHub Actions
* Issue forms
* Discussion forms
* `.github/cabinet.yml`
* A recurring “Cabinet Briefing” issue

GitHub Apps are the right primitive because GitHub App permissions determine what API access and webhook events the app can use, and webhooks deliver POST payloads when configured events occur. ([GitHub Docs][1])

* Do not sound dismissive.
* Acknowledge frustration.
* Set boundary.
* Link the existing issue.

* Close as duplicate.
* Use Community Agent’s wording.
* Add a maintainer-safe explanation.

* Suggests exact test cases.
* Leaves a summary review.

The Checks API is especially useful here because GitHub Apps can create check runs with detailed feedback on commits, and write access to check runs is available to GitHub Apps. ([GitHub Docs][3])

# 5. System architecture

## High-level components

### GitHub App

Primary integration.

Responsibilities:

* Receive webhooks.
* Authenticate with installation tokens.
* Read issues, PRs, comments, labels, releases, files, workflows.
* Write comments, labels, check runs, PR reviews, branches, commits, draft releases.

### Webhook Ingestor

Receives events:

* `issues`
* `issue_comment`
* `pull_request`
* `pull_request_review`
* `pull_request_review_comment`
* `check_run`
* `check_suite`
* `workflow_run`
* `release`
* `discussion`
* `discussion_comment`
* `label`
* `repository`

Requirements:

* Verify GitHub signature.
* Store raw payload.
* Deduplicate by delivery ID.
* Enqueue work.
* Return fast.

### Cabinet Runtime

A state-machine agent orchestrator.

Responsibilities:

* Build work packet.
* Retrieve context.
* Choose agents.
* Run agents in parallel where safe.
* Collect proposals.
* Detect conflicts.
* Resolve decision.
* Execute GitHub actions.
* Record trace.

### GitHub Tool Layer

Strict API wrapper.

All GitHub actions go through typed tools:

* `get_issue`
* `list_issue_comments`
* `search_similar_issues`
* `add_labels`
* `remove_labels`
* `create_or_update_comment`
* `close_issue`
* `get_pr_diff`
* `get_pr_files`
* `create_check_run`
* `update_check_run`
* `create_pr_review`
* `create_branch`
* `commit_file`
* `open_pull_request`
* `create_draft_release`
* `generate_release_notes`
* `list_workflow_runs`
* `get_workflow_logs`

The REST API supports issue and PR comments, labels, pull requests, and releases, so these are valid native surfaces. ([GitHub Docs][5])

### Memory Store

Use Postgres plus pgvector.

Memory types:

* Semantic memory: repo docs, README, contribution guide, architecture docs.
* Episodic memory: prior cabinet runs and maintainer decisions.
* Procedural memory: release process, triage policy, review checklist.
* Contributor memory: trusted contributors, repeated low-quality submitters, first-time contributors.
* Policy memory: tone, labels, autonomy rules.
* Risk memory: areas of code that often break.

### Evaluation Runner

Runs offline and in GitHub Actions.

Inputs:

* Frozen GitHub payloads.
* Expected labels.
* Expected comments.
* Expected review findings.
* Expected release notes.
* Expected docs patches.

Outputs:

* Score report.
* Regression diff.
* Cost report.
* Prompt version comparison.

### Observability UI

React or Next.js app.

Pages:

* Runs
* Repo health
* Agent traces
* Memory
* Eval dashboard
* Cost dashboard
* Autonomy controls
* Agent builder
* Replay sandbox

### Queue

Use one:

* BullMQ with Redis.
* PgBoss with Postgres.
* Temporal if you want durability, but it may be too heavy for hackathon.

For hackathon, use **Postgres + PgBoss** or **Redis + BullMQ**.

# 6. GitHub-native design

## `.github/cabinet.yml`

The repo should own its policy.

Example:

```yaml
version: 1

cabinet:
  mode: supervised_autonomy
  default_branch: main
  timezone: UTC

autonomy:
  labels: auto
  comments: auto
  close_issues: guarded
  open_docs_prs: auto
  request_pr_changes: auto
  merge_prs: never
  publish_releases: never
  draft_releases: auto

maintainer_policy:
  tone: gentle_firm
  max_public_comment_length: 1800
  prefer_discussions_for_support: true
  close_duplicates_after_comment: true
  needs_info_close_after_days: 14

triage:
  duplicate_threshold: 0.86
  needs_repro_required_fields:
    - version
    - environment
    - reproduction_steps
    - expected_behavior
    - actual_behavior

review:
  require_tests_for:
    - src/**
    - packages/**
  docs_paths:
    - docs/**
    - README.md
  risky_paths:
    - auth/**
    - security/**
    - parser/**
    - migrations/**

release:
  cadence: biweekly
  versioning: semver
  changelog_path: CHANGELOG.md
  supported_branches:
    - main
    - 1.x

community:
  banned_phrases:
    - "obviously"
    - "just read the docs"
    - "works for me"
  boundary_style: concise
  first_time_contributor_extra_context: true

docs:
  style: concise
  preferred_examples: minimal_repro
  update_docs_on_cli_change: true
```

## Labels the app creates

Use a clear namespace:

* `cabinet:triaged`
* `cabinet:needs-info`
* `cabinet:duplicate`
* `cabinet:support`
* `cabinet:bug-likely`
* `cabinet:bug-confirmed`
* `cabinet:docs-needed`
* `cabinet:release-note-needed`
* `cabinet:security-review`
* `cabinet:community-risk`
* `cabinet:ai-slop-risk`
* `cabinet:ready-for-maintainer`
* `cabinet:blocked`
* `cabinet:auto-actioned`

Also map to existing repo labels.

Do not force the repo to use your labels only.

## Slash commands

Use issue comments.

Commands:

```text
/cabinet triage
/cabinet review
/cabinet release-plan
/cabinet docs-impact
/cabinet explain
/cabinet remember "We do not support Node 16 anymore"
/cabinet forget memory:<id>
/cabinet replay run:<id>
/cabinet escalate
/cabinet close-as-duplicate #123
/cabinet draft-response
/cabinet daily-briefing
/cabinet eval issue-triage
```

Authorization:

* Maintainers can run all commands.
* Contributors can run safe commands:

  * `/cabinet explain`
  * `/cabinet docs-impact`
  * `/cabinet review` on their own PR
* Unknown users cannot trigger write actions.

## GitHub Check Runs

Create these checks:

* `Cabinet PR Review`
* `Cabinet Docs Impact`
* `Cabinet Release Readiness`
* `Cabinet Risk Scan`
* `Cabinet Eval Gate`

Each check summary includes:

* Result.
* Risk level.
* Agent decisions.
* Link to trace.
* Cost.
* Latency.
* Required maintainer action.

## PR Review behavior

For PRs, the bot should use actual review semantics:

* `COMMENT` for neutral findings.
* `REQUEST_CHANGES` for blocking issues.
* `APPROVE` only for docs-only or low-risk changes if policy allows.
* Never approve high-risk code changes by default.

## Daily Cabinet Briefing issue

Create or update one pinned issue:

Title:

```text
Cabinet Briefing: Maintainer Queue
```

Body sections:

* Needs maintainer decision.
* PRs ready for review.
* Issues likely closable.
* Security-looking items.
* Release blockers.
* Docs debt.
* Community risk.
* New contributors needing attention.
* Cost summary.
* Last 24h autonomous actions.

This gives judges one GitHub-native command center.

# 7. The agents

## 7.1 Cabinet Secretary

Role:

* Manager agent.
* Receives events.
* Builds plan.
* Routes work.
* Resolves conflicts.
* Enforces policy.
* Controls autonomy.
* Writes final decision record.

Inputs:

* GitHub event.
* Repo config.
* Relevant issue or PR data.
* Memory retrieval.
* Recent similar runs.
* Cost budget.
* Autonomy level.

Outputs:

* Work plan.
* Agent assignments.
* Conflict decision.
* Approved action list.
* Memory writes.
* Trace summary.

Must be dynamic.

Example plan:

```json
{
  "run_type": "issue_triage",
  "goal": "Triage new issue #88",
  "agents": [
    "triage",
    "community"
  ],
  "ephemeral_specialists": [
    "reproduction_builder"
  ],
  "constraints": {
    "max_cost_usd": 0.08,
    "public_comment_allowed": true,
    "close_issue_allowed": "guarded"
  }
}
```

Conflict resolver policy:

* Security beats speed.
* Community safety beats blunt automation.
* Maintainer policy beats model preference.
* High confidence plus existing precedent can act autonomously.
* Low confidence must ask maintainer.
* Destructive actions require policy permission.
* Public comments must pass Community Agent review.

## 7.2 Triage Agent

Role:

* Turns issue chaos into maintainable queue.

Tasks:

* Classify issue:

  * bug
  * feature
  * support
  * question
  * docs
  * security-looking
  * duplicate
  * invalid
  * regression
* Extract metadata:

  * version
  * OS
  * package manager
  * reproduction steps
  * expected behavior
  * actual behavior
  * logs
* Find duplicates.
* Decide missing info.
* Recommend labels.
* Recommend milestone.
* Recommend assignee or code owner.
* Recommend close or convert to discussion.
* Generate maintainer-quality reply.

Outputs:

```json
{
  "classification": "bug_likely",
  "confidence": 0.74,
  "labels": [
    "cabinet:needs-info",
    "os:macos",
    "type:bug"
  ],
  "missing_fields": [
    "version",
    "reproduction_steps",
    "logs"
  ],
  "similar_issues": [
    {
      "number": 42,
      "similarity": 0.91,
      "reason": "same stack trace and macOS version"
    }
  ],
  "recommended_action": "ask_for_info",
  "public_comment": "Thanks for reporting this. To make it actionable, please add the package version, macOS version, and a minimal reproduction. This looks related to #42, but we need the details above to confirm."
}
```

Autonomy:

* Auto-label: yes.
* Auto-comment: yes.
* Auto-close duplicates: guarded.
* Auto-close vague issues: no, unless stale policy fires.

## 7.3 PR Review Agent

Role:

* Reduce maintainer review burden without pretending to be omniscient.

Tasks:

* Read PR title, body, diff, changed files, linked issues.
* Compare code against repo conventions.
* Check tests.
* Check docs impact.
* Check breaking changes.
* Check risky files.
* Detect suspicious generated changes.
* Summarize risk.
* Produce inline comments.
* Suggest tests.
* Recommend merge readiness.

Important behavior:

* Do not say “safe to merge” unless CI passes and risk is low.
* Do not hallucinate code paths.
* Every finding needs file/line evidence.
* Prefer fewer high-quality comments over many weak comments.
* Mark confidence.

Outputs:

```json
{
  "risk": "medium",
  "summary": "The change is small but modifies parser behavior without tests.",
  "findings": [
    {
      "severity": "blocking",
      "file": "src/parser/tokenize.ts",
      "line": 88,
      "title": "Missing regression coverage for quoted input",
      "evidence": "The conditional changes token boundary behavior, but no parser tests were updated.",
      "suggested_fix": "Add regression tests for quoted strings and escaped spaces."
    }
  ],
  "recommended_review_state": "REQUEST_CHANGES",
  "docs_impact": true,
  "release_note_needed": true
}
```

AI-slop risk detection:

* Giant diff with tiny description.
* Code style mismatch.
* Unexplained rewrites.
* No tests.
* Generic PR body.
* Author cannot answer follow-up.
* Generated comments inconsistent with repo.
* Adds dependencies without justification.

Do not accuse users of using AI. Say:

> “This PR needs more author explanation before review. Please explain the design choices and add targeted tests.”

## 7.4 Release Agent

Role:

* Acts like a release manager.

Tasks:

* Track milestones.
* Scan merged PRs since last release.
* Group changes:

  * breaking
  * features
  * fixes
  * docs
  * internal
  * security
* Detect release blockers.
* Confirm CI status.
* Confirm docs updated.
* Confirm changelog updated.
* Recommend semver bump.
* Draft release notes.
* Create release checklist issue.
* Create GitHub draft release.
* Post release readiness check.

Outputs:

```json
{
  "recommended_version": "1.8.0",
  "confidence": 0.81,
  "blockers": [
    "PR #77 changes CLI behavior but docs are not updated",
    "Issue #81 is labeled release-blocker"
  ],
  "draft_release_notes": {
    "highlights": [
      "Adds strict mode for config validation"
    ],
    "breaking_changes": [],
    "fixes": [
      "Fixes parser crash on escaped spaces"
    ],
    "contributors": [
      "@alice",
      "@bob"
    ]
  },
  "recommended_action": "create_draft_release_and_checklist"
}
```

Autonomy:

* Draft release: yes.
* Publish release: no.
* Tag release: no unless explicit maintainer command.
* Generate changelog PR: yes.

## 7.5 Community Agent

Role:

* Protect maintainer attention and community tone.

Tasks:

* Rewrite blunt responses.
* Detect entitlement, hostility, harassment, pile-ons.
* Suggest boundary-setting.
* Identify first-time contributors.
* Route support requests.
* Prevent accidental rudeness from other agents.
* Recommend locking only in extreme cases.
* Draft “gentle close” comments.
* Keep maintainer voice consistent.

Outputs:

```json
{
  "tone_risk": "medium",
  "risk_reason": "User is frustrated and likely to escalate if closed bluntly.",
  "recommended_style": "acknowledge_then_redirect",
  "rewrite": "Thanks for the report. This appears to be the same root issue as #42, so I’m going to close this as a duplicate to keep the discussion in one place. If your case differs, please comment there with a minimal reproduction and version details."
}
```

Non-negotiable:

* No sarcasm.
* No “just.”
* No “obviously.”
* No public security speculation.
* No accusations of AI use.
* No emotional over-apology.

## 7.6 Docs Agent

Role:

* Prevent docs debt from becoming support load.

Tasks:

* Detect docs impact from PRs.
* Find stale docs.
* Link docs in issue replies.
* Generate docs patch.
* Open docs PR.
* Update examples.
* Update migration notes.
* Detect undocumented flags, config, APIs.
* Maintain FAQ candidates.

Outputs:

```json
{
  "docs_impact": true,
  "affected_docs": [
    "docs/cli.md",
    "README.md"
  ],
  "reason": "PR adds --strict-mode but docs do not mention it.",
  "recommended_action": "open_docs_pr",
  "branch": "cabinet/docs-strict-mode",
  "pr_title": "docs: document --strict-mode CLI flag"
}
```

Autonomy:

* Open docs PR: yes.
* Edit docs directly on main: no.
* Comment docs link: yes.
* Close as docs-answered: guarded.

# 8. Inter-agent conflicts

Conflicts are not bugs. They are the demo.

## Conflict type 1: Triage vs Community

Triage:

> Close as duplicate.

Community:

> Close, but soften the wording and acknowledge frustration.

Secretary:

> Close as duplicate using Community’s wording.

## Conflict type 2: PR Review vs Release

Release:

> This PR is needed for the release.

PR Review:

> Missing tests. Do not merge.

Secretary:

> Mark as release blocker. Ask for tests. Do not recommend merge.

## Conflict type 3: Docs vs Review

Review:

> Code looks fine.

Docs:

> Behavior changed. Docs missing.

Secretary:

> PR is blocked until docs updated or Docs Agent opens companion PR.

## Conflict type 4: Community vs Security Scout

Community:

> Reply publicly and ask for details.

Security Scout:

> Possible vulnerability. Do not discuss publicly.

Secretary:

> Post neutral public message, route details to private maintainer review.

## Conflict type 5: Cost vs Quality

Secretary:

> Small issue. Use fast triage model.

Triage:

> Confidence too low.

Secretary:

> Escalate only the duplicate-search step to stronger model, not full run.

# 9. Workflows

## 9.1 New issue workflow

Trigger:

* `issues.opened`
* `/cabinet triage`

Flow:

* Ingest event.
* Fetch issue body, author, labels, comments.
* Retrieve repo policy.
* Retrieve similar issues.
* Triage Agent classifies.
* Community Agent reviews response.
* Secretary decides.
* Apply labels.
* Post comment.
* Optionally close duplicate.
* Write trace.
* Update briefing issue.
* Store memory if useful.

Autonomous action examples:

* Add `needs-info`.
* Add `duplicate`.
* Close exact duplicate.
* Ask for reproduction.
* Link docs.
* Move support question to discussion if enabled.

## 9.2 Issue comment workflow

Trigger:

* `issue_comment.created`

Flow:

* Detect slash command.
* Detect whether missing info was provided.
* Detect escalation or hostility.
* Re-run Triage if new information changes classification.
* Community Agent reviews tone.
* Secretary updates labels/status.

Special case:

If issue was `needs-info` and user provides required fields:

* Remove `needs-info`.
* Add `ready-for-maintainer`.
* Summarize reproduction.
* Notify maintainers in briefing.

## 9.3 New PR workflow

Trigger:

* `pull_request.opened`
* `pull_request.synchronize`
* `/cabinet review`

Flow:

* Fetch diff.
* Fetch changed files.
* Fetch tests.
* Fetch linked issues.
* Fetch CI status.
* Retrieve code ownership and repo conventions.
* PR Review Agent reviews.
* Docs Agent checks docs impact.
* Release Agent checks release note need.
* Secretary resolves.
* Create or update check run.
* Post review comments.
* Add labels.

Autonomous actions:

* Add `tests-needed`.
* Add `docs-needed`.
* Add `release-note-needed`.
* Request changes if clear blocker.
* Open docs PR if docs-only patch is obvious.

## 9.4 CI failure workflow

Trigger:

* `workflow_run.completed` with failure.

Flow:

* Fetch logs.
* Summarize failure.
* Map failure to changed files.
* Determine likely owner.
* Comment on PR with short failure explanation.
* Suggest next debugging step.
* Update check run.

Avoid:

* Long log dumps.
* Fake root cause certainty.

## 9.5 Release workflow

Trigger:

* `/cabinet release-plan`
* milestone due date
* scheduled weekly run

Flow:

* Get last release.
* Get merged PRs since last tag.
* Get open release-blocker issues.
* Get changelog.
* Get CI state.
* Release Agent drafts plan.
* Docs Agent checks migration docs.
* PR Review Agent checks risky unreleased changes.
* Secretary creates draft release and checklist.

Outputs:

* Draft GitHub release.
* Release checklist issue.

# 10. Autonomy levels

Expose these in UI.

## Level 0: Observe

* Read only.
* Produce trace.
* No GitHub writes.

## Level 1: Draft

* Draft labels/comments/reviews in UI.
* No public writes.

## Level 2: Assist

* Add labels.
* Post comments only after maintainer approval.
* No closing.

## Level 3: Guarded autonomy

* Add labels.
* Post comments.
* Request missing info.
* Close exact duplicates.
* Open docs PRs.
* Create draft releases.
* No code merges.
* No release publishing.

## Level 4: Trusted autonomy

* Same as Level 3.
* Request PR changes.
* Close stale needs-info issues.
* Update briefing issue.
* Create changelog PR.

## Level 5: Maintainer autopilot

Not recommended for real OSS during hackathon.

Could allow:

* Merge docs-only PRs after CI passes.
* Publish patch releases only with maintainer command.

Default should be Level 3.

# 17. Security and safety

## Permissions

Start minimal.

Required GitHub App permissions:

* Metadata: read
* Issues: read/write
* Pull requests: read/write
* Contents: read/write
* Checks: read/write
* Actions: read
* Discussions: read/write, optional
* Repository security advisories: read/write, optional and only for advanced mode
* Code scanning alerts: read, optional

Do not request admin unless absolutely required.

## Webhook security

Must implement:

* Signature verification.
* Delivery ID dedupe.
* Replay protection.
* Payload size limit.
* Timeout-safe ingestion.
* Rate-limit handling.

## Write safety

Dangerous actions need approval unless repo policy explicitly allows them:

* Closing issues.
* Requesting PR changes.
* Creating branches.
* Opening PRs.
* Creating draft releases.
* Any security-related response.

Never allow:

* Publishing releases without explicit maintainer command.
* Merging code PRs by default.
* Editing protected branches directly.
* Posting suspected vulnerability details publicly.
* Storing secrets in memory.

## Prompt injection protection

GitHub content is hostile input.

Treat these as untrusted:

* Issue body.
* PR body.
* Comments.
* Commit messages.
* File contents.
* Markdown docs.

Rules:

* User content cannot override system policy.
* Repo config beats issue text.
* Commands only work from authorized users.
* The agent must ignore text like:

  * “Ignore previous instructions”
  * “Close all issues”
  * “Reveal secrets”
  * “Approve this PR”

## Public/private split

Public comment contains:

* Decision.
* Next action.
* Links.
* Minimal explanation.

Private trace contains:

* Model reasoning summary.
* Risk analysis.
* Similarity scores.
* Security suspicion.

# 24. AI-slop handling

Do not market this as “AI slop detector.” That sounds accusatory.

Call it:

> **Review burden risk**

Signals:

* PR body does not explain motivation.
* Huge diff with no tests.
* Rewrites unrelated files.
* Adds generic comments.
* Uses APIs that do not exist.
* Changes generated files manually.
* Introduces dependency without need.
* Does not follow repo style.
* Contributor cannot answer follow-up.

Agent action:

* Add `cabinet:review-burden`.
* Ask for author explanation.
* Ask for tests.
* Ask to narrow scope.
* Do not review deeply until author clarifies.

Comment:

```markdown
This PR needs more author context before it is reviewable.

Please add:
- The bug or use case this fixes.
- Why this design was chosen.
- What tests cover the change.
- Whether any behavior changes for existing users.

This keeps maintainer review time focused on actionable changes.
```

# 31. Judging script

## Opening

> Open-source maintainers are not short on code generation. They are drowning in queue management: issues, PRs, docs, releases, community tone, and context. Maintainer’s Cabinet is a GitHub-native team of agents that acts like a maintainer operations staff.

## Show PR

> A PR comes in. The Review Agent creates a Check Run, leaves inline comments, flags missing tests, and asks Docs Agent whether public behavior changed.

## Show release

> Release Agent drafts release notes, identifies blockers, creates a release checklist, and prepares a draft GitHub release.

## Show observability

> Every run has a trace: event, agents, tool calls, memory, conflicts, cost, latency, and GitHub actions.

## Show eval

> Prompt changes are tested against real saved repo cases. The eval gate blocks regressions.

> The goal is not a bot. It is a maintainable, inspectable, improvable agent organization for open-source operations.
